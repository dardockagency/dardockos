# Prompt Lovable — Dashboard V2 Dardock

```text
Quiero que mejores y reconstruyas la V2 de un dashboard-auditoría para Dardock.

IMPORTANTE
La versión actual va bien encaminada, pero todavía se siente como una demo correcta, no como un entregable premium final.

Tu tarea NO es rehacer todo sin criterio.
Tu tarea es corregir los problemas específicos de UX/UI y jerarquía visual para que el dashboard se convierta en una pieza ejecutiva seria.

CONTEXTO
Dardock es una agencia boutique de Growth Partner + Performance Ads.
Este dashboard no es una app SaaS.
No es una herramienta interna de software.
Es un entregable de auditoría ejecutiva para clientes.

Debe sentirse como:
- control room estratégico
- scorecard de performance
- auditoría viva
- interfaz premium de decisión

NO debe sentirse como:
- CRM
- panel de analytics genérico
- app SaaS tradicional
- dashboard corporativo viejo

OBJETIVO DEL DASHBOARD
Responder en segundos:
1. Qué tan sano está el sistema del cliente
2. Qué está funcionando
3. Qué está roto
4. Qué se debe hacer ahora

CRÍTICA DE LA VERSIÓN ACTUAL
Debes corregir específicamente estos problemas:

1. El hero todavía es débil
- le falta tensión visual
- el insight principal no domina suficiente
- hay demasiado aire vacío

2. El score está aislado
- el gauge existe, pero no pesa lo suficiente
- parece decorativo, no central

3. Las KPI cards están correctas pero planas
- todas tienen el mismo peso visual
- falta una métrica protagonista
- falta más sofisticación en contraste y spacing

4. El bloque de capas del sistema está bien pensado pero poco memorable
- necesita más jerarquía
- más respiración
- más claridad en estados

5. Falta narrativa ejecutiva
- debe existir un bloque muy claro de:
  - qué funciona
  - qué necesita atención
  - qué hacemos ahora

6. El dashboard todavía parece más “visual” que “decisorio”
- debe ayudar a decidir, no solo mostrar

7. Debes asumir que el branding técnico final debe quedar listo
- no dejar meta tags genéricos
- no dejar nombres de app genéricos

DIRECCIÓN VISUAL
Mantén la estética dark premium, pero elévala.

Quiero:
- fondo oscuro elegante
- grid sutil
- verde brillante como color de performance
- acentos ámbar para warning
- rojo suave para crítico
- blanco limpio para texto principal
- contraste fuerte pero sofisticado

Debe sentirse:
- técnico
- premium
- ejecutivo
- moderno
- con mucha autoridad visual

TIPOGRAFÍA
- títulos con mucho peso y presencia
- cuerpo limpio y muy legible
- excelente jerarquía
- tono editorial + dashboard

ESTRUCTURA OBLIGATORIA V2

1. HERO EJECUTIVO
Debe contener:
- nombre del cliente
- rubro / descriptor del negocio
- periodo auditado
- insight principal MUY visible
- score general
- estado general
- microfrase de interpretación

Ejemplo de insight:
- “Sistema rentable, pero inestable”
- “La adquisición funciona. El funnel no.”
- “Buen tráfico. Señal incompleta.”

El insight principal debe pesar más que el nombre del cliente.

El score debe incluir:
- número grande
- etiqueta de estado
- texto tipo:
  - “3 capas requieren corrección antes de escalar”
  - “Tracking y funnel limitan la eficiencia real”

2. KPI SECTION
No quiero 6 cards iguales.

Quiero:
- 1 KPI protagonista
- 4 o 5 KPIs secundarios

Ejemplo:
- KPI protagonista: ROAS o CPA
- secundarios: inversión, CVR, frecuencia, ticket, revenue atribuido

Cada KPI debe incluir:
- valor
- comparación
- estado visual

3. ESTADO DEL SISTEMA POR CAPAS
Las 5 capas:
- Oferta
- Tracking
- Funnel
- Campañas
- Creativo

Cada card debe tener:
- score
- estado
- comentario corto
- barra o indicador

Agrega arriba un resumen visual tipo:
- 2 capas críticas
- 2 en observación
- 1 estable

4. BLOQUE “QUÉ FUNCIONA / QUÉ NECESITA ATENCIÓN”
Este bloque es obligatorio.

Debe tener dos módulos bien diferenciados:

A. Qué funciona
- 3 a 5 fortalezas reales

B. Qué necesita atención
- 3 a 5 fugas o debilidades reales

5. BLOQUE DE SÍNTOMAS Y CAUSAS
Debe mostrar:
- síntoma
- causa probable
- capa afectada
- impacto

Formato:
- cards compactas
- o tabla muy limpia

6. ROADMAP
Debe verse con peso visual, no como un pie de página.

Dividir en:
- urgente
- 30 días
- 60-90 días

Cada acción debe incluir:
- título
- impacto esperado
- dependencia si aplica

7. PRÓXIMOS PASOS
Separar:
- Qué hará Dardock
- Qué debe hacer el cliente

CASO DE DEMO
Ya no quiero un caso genérico estilo “NovaSkin”.

Quiero que el template se sienta preparado para un caso real como:
- ecommerce
- Meta + Google + Shopify
- problemas de tracking, funnel, creative fatigue, estructura, etc.

No hace falta meter toda la data real todavía, pero sí quiero que la demo sea más creíble y más cercana a un caso como Cosas Nuestras.

UX/UI DETALLADO

Quiero:
- mucho mejor uso del espacio en el hero
- mejor balance entre insight principal y score
- cards más refinadas
- mejor ritmo vertical
- más contraste entre módulos
- módulos con distinta intensidad visual según prioridad

No quiero:
- demasiado aire muerto
- módulos con el mismo peso
- sensación de wireframe bonito
- look de admin dashboard genérico

RESPONSIVE
- impecable en desktop
- bien resuelto en tablet
- funcional y claro en mobile

OUTPUT ESPERADO
Quiero una V2 que se sienta como un entregable premium de Dardock.
Debe verse más como:
- auditoría estratégica visual
y menos como:
- dashboard estándar de app

Además:
- actualiza título y meta tags reales
- elimina branding genérico de Lovable
- deja el resultado listo para iterar hacia un caso real
```
